De LLMs a Agentes: Entiende, Construye y Automatiza con IA

Clase 1 — Curso de Inteligencia Artificial para Programadores. Del prompt al producto: hoy recorremos el camino completo desde entender como funciona un LLM hasta convertirlo en un agente autonomo que trabaja para vos.

02 · Apertura

La gran pregunta

Si un LLM no piensa, no razona y no entiende... ¿como es posible que pueda escribir codigo funcional? ¿Como diagnostica errores? ¿Como genera soluciones que a veces son mejores que las nuestras? Esa tension entre lo que el modelo NO es y lo que PARECE ser es exactamente lo que vamos a explorar hoy.

03 · Apertura

Mapa de la clase

Hoy recorremos 5 bloques: un repaso rapido de LLMs, prompting profesional, RAG para darle memoria externa, el salto de chatbot a agente, y el stack moderno de herramientas. Cada bloque construye sobre el anterior.

04 · Repaso LLM

Lo que ya sabemos

Texto entra, se parte en tokens, se transforma en numeros, pasa por capas de atencion, y al final el modelo genera una probabilidad para cada posible siguiente token. Eso es TODO lo que hace. Tres ideas clave: parametros = conocimiento congelado, ventana de contexto = memoria RAM, alucinaciones = no tiene "no se".

05 · Repaso LLM

Lo que falta

Saber que ES un LLM no es lo mismo que saber USARLO profesionalmente. Hoy vamos a cerrar esa brecha con tres preguntas: ¿como le hablo al modelo? ¿como le doy informacion que no tiene? ¿como lo convierto en un agente autonomo?

06 · Prompting

El prompt es codigo

Un prompt NO es una pregunta. Es una instruccion estructurada que define el comportamiento del modelo. Analogia: un prompt es como los parametros de una funcion. Si tu funcion recibe basura, devuelve basura. Con un prompt es exactamente igual.

07 · Prompting

Anatomia de un prompt profesional

Un prompt profesional tiene 5 capas, como una funcion con parametros tipados: System Prompt (quien es), Contexto (que necesita saber), Instruccion (que hacer), Formato (como responder), y Ejemplos (few-shot learning).

08 · Prompting

System Prompt en profundidad

El system prompt es el ADN del agente. Define QUIEN es el modelo: su rol, su tono, sus restricciones y su formato de respuesta. Es como el constructor de una clase: se ejecuta una vez y configura todo el comportamiento posterior.

09 · Prompting

Tecnicas de prompting

Cuatro tecnicas que van a usar constantemente: Zero-shot (instruccion directa), Few-shot (con ejemplos), Chain of Thought (razonamiento paso a paso), y Role Playing (asignar identidad). Cada una tiene su momento.

10 · Demo en vivo

Prompt malo vs profesional

Comparacion en vivo: a la izquierda un prompt vago ("hazme una API REST"), a la derecha un prompt profesional con system prompt, contexto, formato y restricciones. La diferencia en la salida es dramatica. El prompt es codigo — tratalo como tal.

11 · Prompting

Reglas en Markdown

Cuando trabajas con agentes de codigo, las reglas del sistema no van en un chat. Van en archivos .rules.md que viven en el repositorio. Eso te permite versionar el comportamiento del agente con Git, como cualquier otra parte del codigo.

12 · Prompting

El prompt como contrato

Si tu prompt es vago, la respuesta sera vaga. Si tu prompt es preciso, la respuesta es predecible. Regla de oro: trata cada prompt como una funcion con tipado estricto. Definí entradas, salidas y restricciones.

13 · Prompting

Errores comunes en prompting

Los 5 errores que mas veo: no definir el rol, pedir todo en un solo mensaje, no especificar formato de salida, no iterar sobre el resultado, e ignorar el system prompt. Evitalos y tus resultados mejoran inmediatamente.

14 · RAG

El problema: el modelo alucina

¿Que pasa si le preguntas al modelo sobre TU codigo? ¿Sobre la documentacion interna de TU empresa? ¿Sobre datos que se generaron ayer? Alucina. Inventa. Porque no tiene esa informacion. Fue entrenado hace meses con datos publicos.

15 · RAG

La solucion: RAG

Retrieval Augmented Generation: un patron de arquitectura que inyecta informacion relevante al modelo EN EL MOMENTO de la consulta. Tus documentos se indexan, se buscan por similitud, y los fragmentos relevantes se pasan junto con la pregunta.

16 · RAG

Embeddings explicados

Un embedding convierte texto en un vector numerico que captura su significado. Palabras similares ("perro", "gato", "mascota") quedan cerca en el espacio vectorial. Palabras distintas ("economia", "galaxia") quedan lejos. Eso es lo que permite la busqueda por similitud.

17 · RAG

RAG paso a paso

Tres fases: (1) Indexacion: documentos se parten en chunks, se convierten en embeddings y se guardan en vector DB. (2) Consulta: la pregunta se convierte en embedding, se buscan los chunks mas similares. (3) Generacion: chunks + pregunta van al LLM que genera respuesta con datos reales.

18 · RAG

Cuando usar RAG

Usa RAG cuando tengas documentacion interna, datos privados, codigo propietario o informacion que se actualiza. NO lo uses para conocimiento general, tareas creativas o datos publicos — el modelo ya sabe eso. RAG complementa, no reemplaza el fine-tuning.

19 · Agentes

El espectro de la IA

Desde autocompletado hasta sistemas multi-agente. A la izquierda: Copilot predice la siguiente linea. En el medio: ChatGPT conversa. A la derecha: un agente que planifica, ejecuta, verifica y corrige autonomamente. La diferencia critica: un chatbot espera instrucciones, un agente recibe un OBJETIVO.

20 · Agentes

Chatbot vs Agente

Un chatbot RESPONDE preguntas. Un agente HACE cosas. Sin memoria persistente vs con memoria entre sesiones. Sin herramientas vs con acceso a filesystem, APIs, bases de datos. Reactivo vs proactivo. La diferencia es fundamental.

21 · Agentes

Anatomia de un agente

Un agente tiene 4 componentes: el LLM como cerebro que razona, las herramientas como manos que ejecutan, la memoria como experiencia acumulada, y la planificacion como estrategia. Los cuatro trabajan en ciclo continuo.

22 · Agentes

MCPs — Model Context Protocol

Los MCPs son el USB-C de los agentes: un estandar para conectar cualquier herramienta. Antes, cada integracion era custom. Con MCP, hay un protocolo universal. Un agente con soporte MCP puede conectarse a GitHub, Slack, bases de datos, filesystem, cualquier cosa.

23 · Agentes

MCPs en la practica

Ejemplo real: un agente con MCP de GitHub puede leer issues, crear PRs y revisar codigo. El flujo: el usuario pide algo, el agente planifica, llama al MCP server, ejecuta la accion, y reporta el resultado. Y lo mejor: podes crear tus propios servidores MCP.

24 · Agentes

SKILLs — Capacidades modulares

Un SKILL es un paquete de instrucciones + herramientas que le da al agente una capacidad especifica. Como un plugin con cerebro. Ejemplo: SKILL de "crear documentos DOCX" incluye instrucciones, templates y validaciones. Se cargan bajo demanda, no todo junto.

25 · Agentes

Engram — Memoria persistente

Por defecto, un agente olvida todo entre conversaciones. Engram resuelve eso: guarda contexto relevante (decisiones, bugs, convenciones) y lo recupera automaticamente cuando es necesario. Es la diferencia entre un agente amnesico y uno que aprende.

26 · Agentes

El loop del agente

Todo agente sigue un ciclo: Percibir (recibe input), Pensar (razona sobre que hacer), Actuar (usa herramientas), Observar (evalua el resultado). Repite hasta completar la tarea. Es el patron fundamental.

27 · Demo en vivo

Agente en accion

Demo: le damos un objetivo a un agente (Claude Code u OpenCode): "Crea un endpoint REST en Express con TypeScript que valide emails con Zod. Incluye tests." Observamos como planifica, escribe codigo, ejecuta tests y corrige errores. Nadie le dice paso a paso — le dimos un OBJETIVO.

28 · Stack Moderno

El nuevo flujo de trabajo

El codigo ya no se escribe solo. Se dirige. Antes: IDE + Stack Overflow + copiar/pegar. Ahora: Spec + Agente + Revision humana. El programador pasa de escritor a director. Pero director informado — tenes que entender lo que dirigis.

29 · Stack Moderno

SDD — Spec Driven Development

Antes de escribir codigo, escribi una spec. Una spec define QUE queres construir: objetivo, contexto, requisitos, restricciones, criterios de aceptacion. La calidad de lo que genera el agente es directamente proporcional a la calidad de la spec.

30 · Stack Moderno

Estructura de una Spec

Una spec tiene: Objetivo (que queres lograr), Contexto tecnico (stack, dependencias), Requisitos funcionales (que debe hacer), Restricciones (que NO debe hacer), y Criterios de aceptacion (como saber que esta bien). Una buena spec = mejor codigo generado.

31 · Stack Moderno

Worktrees — Paralelismo real

Git worktrees te permiten trabajar en multiples features simultaneamente sin conflictos. Cada worktree tiene su propio directorio con su propio agente. Un agente en login, otro en la API, otro arreglando bugs. Paralelismo real.

32 · Stack Moderno

Herramientas del ecosistema

El ecosistema actual: Claude Code (agente de terminal premium), OpenCode (alternativa open source), Cursor y Windsurf (IDEs con IA integrada), GitHub Copilot (autocompletado). Cada una tiene su lugar segun el caso de uso.

33 · Stack Moderno

Gentle AI Stack

Un stack completo y accesible para empezar a construir con agentes. 5 capas: Base (Git + Terminal), Agente (OpenCode/Claude Code), Herramientas (MCPs + SKILLs), Memoria (Engram), Metodologia (SDD + Specs). Cada capa habilita la siguiente.

34 · Stack Moderno

Eleccion de modelo

Para desarrollo local: Ollama + modelos open source (Llama, Mistral, DeepSeek). Privacidad total, sin costo, menor capacidad. Para produccion: Claude, GPT-4o, Gemini. Maxima capacidad, costo por token. Regla: prototipa local, despliega en cloud.

35 · Demo en vivo

El flujo completo

Demo final: (1) Escribimos una spec.md rapida. (2) Se la pasamos a un agente. (3) Vemos como genera el codigo. (4) Revisamos el resultado. El flujo completo de SDD en 3 minutos. Asi se trabaja con IA en proyectos reales.

36 · Cierre

Lo que vimos hoy — 5 ideas clave

(1) Un LLM predice tokens, no piensa. (2) Un prompt profesional es codigo con estructura. (3) RAG elimina alucinaciones con datos reales. (4) Un agente = LLM + herramientas + memoria + planificacion. (5) El futuro es dirigir IA, no solo escribir codigo.

40 · Cierre

¿Preguntas?

La mayoria de los programadores usan IA como una calculadora sofisticada. Pero lo que vimos hoy es que la IA puede ser un COMPAÑERO DE EQUIPO. Uno que lee specs, escribe codigo, corre tests y aprende de sus errores. La pregunta no es SI van a trabajar con agentes. La pregunta es CUANDO empiezan.
